# 优雅降级

Incident Commander 在数据源不可用时优雅降级，而非崩溃或产生空结果。

## 降级链

```text
采集器降级：
  MCP Server ──(不可用)──▶ CLI 工具 ──(不可用)──▶ 引导手动输入 ──(跳过)──▶ 标记数据缺失

整体降级：
  全自动模式 ──(部分源不可用)──▶ 半自动模式（可用源采集 + 手动补充）
              ──(全源不可用)──▶ 纯交互模式（问答采集信息）
```

## 采集器级降级

每个采集器都有降级链。以 GitHub 采集器为例：

```text
步骤 1: 尝试 GitHub MCP Server
  ├── 成功 → 返回事件
  └── 失败 ↓

步骤 2: 尝试 gh CLI
  ├── 成功 → 返回事件
  └── 失败 ↓

步骤 3: 引导手动粘贴
  ├── 用户提供数据 → 解析并返回事件
  └── 用户跳过 ↓

步骤 4: 标记数据缺失
  └── 继续使用其他数据源的事件
```

## 系统级降级

### 全自动模式（所有数据源可用）

```text
/incident start 2h
📊 采集: GitHub ✅ | Sentry ✅ | Grafana ✅
📊 时间线: 25 个事件，4 个转折点
🧠 RCA: 高置信度因果链
📝 Post-Mortem: 完整报告已生成
```

### 半自动模式（部分数据源不可用）

```text
/incident start 2h
📊 采集: GitHub ✅ | Sentry ❌ (认证失败) | Grafana ✅
📊 时间线: 来自 2 个数据源的 17 个事件
🧠 RCA: 中置信度（错误数据有限）
📝 Post-Mortem: 已生成，缺失部分标记为 [MISSING: Sentry 错误详情]
```

### 纯交互模式（所有数据源不可用）

```text
/incident start 2h
📊 采集: 无可用自动数据源
📋 启动交互式问答模式...
  Q: 事件什么时候开始的？
  A: 大约上午 10:00
  Q: 最初的症状是什么？
  A: /api/users 出现 500 错误
  Q: 如何修复的？
  A: 我们在 10:30 回滚了
📊 时间线: 3 个事件（手动提供）
🧠 RCA: 低置信度（问答数据有限）
📝 Post-Mortem: 草稿已生成 — 需要大量编辑
```

## 错误处理

| 场景 | 行为 | 用户提示 |
|------|------|---------|
| MCP 超时 | 降级到 CLI | "MCP 服务器超时，正在降级到 gh CLI..." |
| CLI 未找到 | 引导手动输入 | "未找到 gh CLI，请粘贴你的 GitHub 事件..." |
| 认证失败 | 跳过该数据源 | "Sentry 认证失败，正在跳过 Sentry 数据..." |
| 速率限制 | 带退避重试 | "触发 GitHub API 速率限制，30 秒后重试..." |
| 网络错误 | 降级到缓存 | "网络错误，正在使用上次运行的缓存数据..." |
| 数据无效 | 过滤错误条目 | "已跳过来自 Grafana 的 2 个格式错误事件..." |

## 置信度调整

RCA 置信度根据可用数据自动调整：

| 可用数据源 | 最高置信度 | 原因 |
|-----------|-----------|------|
| 3+ 个数据源 | 🟢 高 | 可进行交叉验证 |
| 2 个数据源 | 🟢 高 | 部分交叉验证 |
| 1 个数据源 | 🟡 中 | 无法交叉验证 |
| 仅手动提供 | 🔴 低 | 基于用户叙述 |

## 最佳实践

1. **至少配置 2 个数据源** — 交叉验证显著提升 RCA 质量
2. **保持 gh CLI 认证** — 它是最可靠的降级方案
3. **查看缺口标记** — Post-Mortem 中的 `[MISSING]` 部分需要手动补充
4. **定期运行 `/incident config`** — 确保数据源保持连接
